用 AI Agent 构建零代码知识库：采集、分析、归档一条龙

从一个问题说起#

每天 GitHub Trending 上都有新的 AI 项目冒出来，Hacker News 上技术讨论层出不穷。面对信息洪流，手动筛选和整理变得越来越不现实。能不能让 AI 替我们把这件事干了——不只是简单抓取，而是真正理解内容、评估价值、结构化存储？

最近做了一个实验性项目 AI Knowledge Base，给出了一个有趣的答案：零行业务代码，纯 Agent 驱动的自动化技术情报系统。

核心思路：声明式 + Agent 协作#

整个系统的设计哲学很简单——用 Markdown 声明意图，让 LLM 充当执行引擎。

传统做法是写 Python 脚本调 API、做清洗、存数据库。这个项目换了个方向：通过角色定义文件（Markdown）描述每个 Agent 的职责、权限和操作规范，LLM 在 OpenCode 框架内根据这些声明自主完成任务。

三阶段流水线#

Collector（采集员）  →  Analyzer（分析员）  →  Organizer（归档员）
      ↓                      ↓                       ↓
 knowledge/raw/        原地 enrich 数据       knowledge/articles/

三个角色，各司其职，单向流动：

Agent做什么能用什么工具绝对不能做什么
Collector从 GitHub / HN 抓数据Read, WebFetchWrite（不能写文件）
Analyzer生成摘要、评分、打标签Read, WebFetchWrite（不能写文件）
Organizer去重、过滤、写入最终条目Read, Write, EditWebFetch（不能再外联）

Collector 只采集不分析，Analyzer 只分析不写入，Organizer 只整理不外联。

这种设计不是随意为之。单向数据流加上最小权限原则，意味着即使某个 Agent 出了问题，影响范围也被严格控制在它的职责边界内。Organizer 没有网络权限，所以它不可能把你的数据泄露出去。Analyzer 不能写文件，所以它不可能污染原始数据。

权限隔离为什么重要#

在很多 AI 应用中，LLM 拥有过于宽泛的能力——既能读又能写还能联网。这在简单场景下没问题，但在多 Agent 协作的复杂系统中，权限泛滥是安全隐患。

这个项目采取了最小权限原则：

  • Collector 需要 WebFetch 访问外部 API，但不应该修改文件系统 → 禁止 Write
  • Analyzer 需要阅读原始数据并可能抓取页面补充信息，但同样不应该写入 → 禁止 Write
  • Organizer 负责最终的文件操作，但此时所有外部数据已就绪 → 禁止 WebFetch

每个 Agent 的权限在角色定义文件中以 YAML 格式显式声明：

# collector.md
allowed-tools:
  - Read
  - Grep
  - Glob
  - WebFetch

LLM 框架在运行时强制执行这些约束，Agent 无法越权。

质量门控：不是什么都要#

自动化最怕的不是效率低，而是信息泛滥。每天可能有上百条 AI 资讯，但大部分是重复的、浅层的、炒作性质的。

Analyzer 采用了一个五维评分体系来量化每条资讯的价值：

relevance_score = 技术深度×0.25 + 实用价值×0.30 + 时效性×0.20 + 社区热度×0.15 + 领域匹配×0.10

每项 0-1 分，加权平均后得到最终得分。Organizer 对 relevance_score < 0.6 的条目直接丢弃。

这个公式背后有明确的优先级：实用价值最重（0.30），技术深度其次（0.25），领域匹配最轻（0.10）。 这反映了项目定位——面向工程师的实战知识库，不是学术论文索引。

评分不是黑盒。每个条目都保留了完整的评分分解：

{
  "relevance_score": 0.87,
  "score_breakdown": {
    "tech_depth": 0.80,
    "practical_value": 0.95,
    "timeliness": 0.90,
    "community_heat": 0.85,
    "domain_match": 0.95
  }
}

如果某条被丢弃，过滤日志会记录原因，人工可以复查。

幂等性与去重#

自动化系统必须可重复运行。跑两次不应该产生两份数据。

项目从两个层面保证幂等性：

  1. 文件命名包含日期：github-trending-2026-03-17.json，同一天的数据覆盖写
  2. Organizer 双重去重：URL 精确匹配 + 标题模糊匹配（相似度 > 90%）

这意味着你可以放心地每天运行，甚至一天运行多次，不会出现重复条目。

可扩展性：加一个数据源有多难？#

如果要添加 arXiv 作为新的数据源，需要改多少代码？

答案是零。只需要写一个 skills/arxiv/SKILL.md，定义采集策略、API 端点、字段映射。Collector 会根据这个声明文件自动适配。Analyzer 和 Organizer 无需任何修改。

这种可扩展性来自于职责的清晰分离。Collector 只关心"怎么采集"，Analyzer 只关心"怎么分析"，Organizer 只关心"怎么整理"。新增数据源不影响分析和整理逻辑。

全链路可追溯#

每个知识条目保留了完整的时间戳链：

{
  "collected_at": "2026-03-17T10:30:00Z",
  "analyzed_at": "2026-03-17T11:00:00Z",
  "organized_at": "2026-03-17T11:30:00Z"
}

从采集到分析到归档，每一步都有记录。如果某条数据有问题，你可以追溯到它从哪个来源来、什么时候被采集、谁分析的、得分如何。这对知识库的长期维护至关重要。

零代码意味着什么#

整个项目没有一行业务代码。没有 Python 脚本，没有数据库，没有服务端。运行时依赖只有：

  • OpenCode 框架（提供 Agent 运行环境）
  • LLM（DeepSeek / Qwen 等国产模型即可）
  • 文件系统（JSON 文件存储）

系统的全部"代码"是 Markdown 声明文件：

v1-skeleton/
├── AGENTS.md                          # 全局规则、编码规范、工作流
├── .opencode/
│   ├── agents/
│   │   ├── collector.md               # 采集员角色定义
│   │   ├── analyzer.md                # 分析员角色定义
│   │   └── organizer.md               # 归档员角色定义
│   └── skills/
│       ├── github-trending/SKILL.md   # GitHub 采集操作手册
│       └── tech-summary/SKILL.md      # 摘要生成写作规范
└── knowledge/
    ├── raw/                           # 原始数据
    └── articles/                      # 最终知识条目 + 索引

Markdown 即代码。 LLM 读取这些声明文件，理解自己的角色和规则，然后执行。这本质上是把传统编程中的"业务逻辑"从代码转移到了自然语言声明中。

这能推广到其他场景吗？#

这种架构模式不限于技术情报收集。任何符合以下特征的场景都可以复用：

  • 多步骤数据处理：采集 → 转换 → 存储
  • 需要质量过滤：不是所有输入都值得保留
  • 数据源多样：需要从不同 API / 网站获取信息
  • 低频但持续：不需要实时响应，定时运行即可

比如：竞品分析、学术文献追踪、行业政策监控、开源 License 合规检查等。

局限与反思#

坦率地说，这种纯声明式方案也有局限：

  1. 运行成本：每次执行都需要 LLM 推理，比传统脚本贵得多
  2. 确定性：LLM 输出不完全确定性，同样的输入可能产生略有差异的结果
  3. 速度：Agent 逐条分析比批量脚本慢，不适合高频实时场景
  4. 调试：当结果不符合预期时，调试"自然语言代码"比调试 Python 更难

这些问题并非无解。可以通过 prompt 工程提高确定性，通过缓存和增量处理降低成本，通过结构化日志辅助调试。但你需要清楚，这是在用灵活性和可维护性换取确定性和效率。

总结#

这个项目展示了 AI Agent 协作的一种可能性：不是让一个大模型包办一切，而是通过角色分工、权限隔离、质量门控来构建一个可控的自动化系统。

几个关键设计值得借鉴：

  • 最小权限原则：每个 Agent 只拥有完成本职工作所需的工具
  • 单向数据流：数据从采集到归档只往前走，不可逆
  • 质量门控：自动过滤低价值内容，避免信息泛滥
  • 幂等设计：重复运行不会产生副作用
  • 声明式驱动：Markdown 即配置，LLM 即运行时

AI 不只是聊天机器人。当你把它当作一个可编排的执行引擎，配合合理的架构约束，它能做的事情远比想象中多。

项目基于 OpenCode 框架，使用 DeepSeek / Qwen 等国产 LLM 作为推理引擎。完整源码和声明文件可在 GitHub 查看。
